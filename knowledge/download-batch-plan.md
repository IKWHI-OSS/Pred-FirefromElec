# 다운로드 배치 계획 — 재설계 (v2)

> 작성 2026-06-14. 정본: 이 파일이 배치 다운로드 운영의 단일 기준.
> 이전 계획("배치당 ~90GB, 6배치")을 **peak(최대 점유) 공간 모델**로 다시 설계함.

## 0. 요약 (TL;DR)

- 원래 규칙 "여유의 절반"은 **틀렸다기보단 위험**. aihubshell은 `download.tar`를 받은 뒤
  **압축해제하는 순간 tar와 추출본이 동시에 디스크에 존재**한다 → 순간 점유 ≈ 배치크기 × 2.
- 따라서 안전 배치 한도는 **`B_max ≈ 가용공간 × 0.40`** (peak 0.80, 여유 20% 버퍼).
- 269GB 여유 기준 → 배치당 최대 ~107GB 가능하나, **안정성 위해 ~80GB 권장**.
- 배치 수는 고정하지 말고 **파일트리 실측 + 실시간 df로 자동 산정**한다(플래너 스크립트 제공).
- **삭제는 GCS 업로드 검증(객체 존재+크기 일치) 통과 후에만** (SOUL.md 파괴적 변경 원칙).

## 1. 공간 모델 (왜 재설계했나)

aihubshell 다운로드 흐름(메모리 도구특성 참조):
```
curl -L -C - 로 download.tar 수신  →  tar 자동 압축해제  →  .part 조각 병합
```
이미지(jpg/png)는 이미 압축돼 있어 `tar ≈ 추출본 크기`. 압축해제 중에는 둘이 공존하므로:

```
peak_usage(배치) ≈ B(tar) + B(추출본) = 2 × B      # 추출 완료 후 tar 삭제 전까지
```

운영 루프가 `다운로드 → 업로드 → 로컬삭제 → 다음배치`이므로, 한 시점의 최대 점유는 한 배치의 peak.
안전식:

```
free ≥ peak + buffer
2 × B + 0.20 × free ≤ free
⇒ B_max = 0.40 × free          (버퍼 20% 확보)
```

| 가용공간(free) | 이론 한도(0.40×free) | 권장(안정) |
|---|---|---|
| 269 GB | 107 GB | ~80 GB |
| 175 GB | 70 GB | ~55 GB |
| 120 GB | 48 GB | ~38 GB |

> ⚠ **현재 가용공간 확인 필요**: 배치1(~94GB)이 *아직 로컬에 있는지(업로드 전)* 여부에 따라
> free가 269 → 175GB로 바뀜. 플래너 스크립트가 `df`를 실시간으로 읽어 자동 반영한다.

## 3. 재설계 전략

1. **고정 배치 수를 버린다.** 파일트리(`aihubshell -mode l`)의 filekey별 실제 크기를 받아,
   `B_max` 이하로 **그리디 빈패킹**하여 배치를 자동 생성한다.
2. **티어 순서**: TS(원천) 먼저 → VS(원천) → 라벨+GPS는 마지막 배치 꼬리에 합침(극소라 공간영향 0).
3. **권장 배치 크기 ~80GB** 고정(269GB free 가정). free가 줄면 플래너가 자동으로 더 작게 쪼갬.
4. **루프 불변식**: 어느 순간에도 로컬엔 *최대 1배치*만 존재. 업로드·검증 후 즉시 삭제.

대략적 결과(269GB free, ~80GB/배치 가정, 실측 전 추정):

| 배치 | 티어 | 추정 크기 | 비고 |
|---|---|---|---|
| 1 | TS | ~94 GB | 완료(504524–504529) |
| 2 | TS | ~80 GB | 플래너가 filekey 확정 |
| 3 | TS | ~80 GB | |
| 4 | TS | ~80 GB | TS 잔여 흡수 |
| 5 | VS | ~68 GB | + 라벨/GPS 합침 |

→ **5배치**로 수렴(이전 6배치에서 축소). 단 **실측 후 플래너 출력이 최종**.

## 4. 운영 절차 (배치 루프)

```
[맥, 한국 IP] 에서:
1) 파일트리 덤프:   aihubshell -mode l -datasetkey 71388 > tree.txt
2) 배치 계획 산정:   python3 plan_batches.py --tree tree.txt --target-dir <다운로드폴더>
3) 각 배치마다:
   a. 다운로드:      aihubshell -mode d -datasetkey 71388 -aihubapikey '키' -filekey <배치 키들>
   b. 업로드:        gsutil -m cp -r <받은폴더> gs://<버킷>/237/<경로>/
   c. 검증:          업로드 객체 존재 + 바이트 크기 일치 확인 (불일치 시 삭제 중단)
   d. 로컬 삭제:      검증 통과 시에만 rm -rf <받은폴더>
   e. 다음 배치
```

## 5. 안전장치 (파괴적 삭제 방지)

- 삭제 전 **반드시** `gsutil ls -l gs://...` 로 객체 존재 + 크기 대조.
- 크기 불일치/누락 시 **삭제 금지**, 재업로드.
- 첫 배치는 삭제 전 사용자 1회 확인(이후 검증 자동화 신뢰되면 무인 루프).
